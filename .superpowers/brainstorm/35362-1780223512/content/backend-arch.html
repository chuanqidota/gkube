后端架构方案对比

选择 gkube 后端的代码组织方式

A

方案 A：在现有结构上扩展（推荐）

保留现有的 app/k8s/ + pkg/k8s/ 三层结构，新增模块沿用相同模式

  • 新增模块：app/auth/、app/dashboard/、app/cluster-mgmt/
  • API 路由：/v1/auth/*、/v1/clusters/*、/v1/dashboard/*
  • 优点：改动最小，与现有代码风格一致，快速交付
  • 缺点：现有结构的一些问题会延续（如 params 和 api 的对应关系不够清晰）
B

方案 B：模块化重构

按业务领域重新组织目录，每个模块内部自包含 handler/service/model

  • 目录结构：internal/cluster/、internal/auth/、internal/resource/、internal/dashboard/
  • 每个模块：api.go + service.go + model.go + params.go
  • 优点：模块边界清晰，便于独立开发和测试
  • 缺点：需要迁移现有代码，工作量较大
C

方案 C：Clean Architecture

采用领域驱动设计，严格的四层架构

  • 四层：domain/ → usecase/ → adapter/ → infrastructure/
  • 优点：架构最清晰，可测试性最强
  • 缺点：学习曲线陡峭，前期投入大，对轻量级项目可能过度设计
